# 구현 로드맵

## 작업 원칙

- 이 문서 작성까지는 상거래 코드를 변경하지 않는다.
- 실제 구현은 현재 문서 변경과 분리된 새 브랜치에서 시작한다.
- 한 작업은 가능한 한 하나의 데이터 계약 또는 사용자 흐름만 변경한다.
- DB 변경은 새 순방향 마이그레이션으로 수행하고 적용된 기존 마이그레이션을 수정하지 않는다.
- 결제·권한·주문·재고·배송 변경은 관련 테스트, 린트, 빌드, 마이그레이션 정합성을 검증한다.
- 운영 환경과 인증이 필요한 흐름은 정적 검사와 별도로 실제 검증 결과를 기록한다.

## P0 — 기준과 거래 안전성

### P0-1. 미결 정책 결정 기록

- [현행 감사](./current-state-audit.md)의 8개 미결 항목을 결정한다.
- 결정마다 데이터/API/UI 영향과 승인 날짜를 기록한다.
- 성공 조건: 이후 스키마가 추측 없이 작성 가능하다.

### P0-2. 수동 계좌이체 단일 활성 경로

상태: 코드 구현 및 로컬 검증 완료, 운영 DB 마이그레이션·배포 대기 (`feature/manual-transfer-only`)

- 고객 UI에서 PortOne 선택과 실행 진입점을 제거한다.
- 서버 체크아웃은 수동 계좌이체만 허용한다.
- PortOne 라우트와 어댑터는 명시적 비활성 상태로 격리·보존한다.
- 운영 중 기능 설정이 우연히 PortOne으로 바뀌지 않도록 서버 제약을 둔다.
- 성공 조건: 수동 주문·입금 원장은 정상이고 PortOne 공개 경로는 실행되지 않으며 관련 코드는 재활성화 가능한 형태로 남는다.

### P0-3. 매장 멤버십과 세부 권한

- `store_memberships` 또는 동등 모델을 추가한다.
- 기존 운영자·직원 데이터를 무손실로 이행한다.
- API와 RLS가 같은 매장 범위를 검사하도록 한다.
- 성공 조건: 운영자와 직원은 허용된 매장만 다루고 권한 있는 직원의 상품 공개가 가능하다.

### P0-4. 상품 작성·즉시 공개 계약

- 외부 승인 의미를 제거한다.
- `draft` 저장과 권한 보유자의 즉시 `active` 공개를 구분한다.
- 검수 메모는 상품 설명 데이터로 유지하고 승인 상태와 분리한다.
- 성공 조건: 승인 대기자 없이 자기 매장 상품을 공개하며 타 매장 수정은 차단된다.

### P0-5. 통합 입금의 공동 조회·멱등 확인

- 확정된 공개 범위로 공용 입금 큐를 제공한다.
- 원장 기록과 주문 결제 확정을 원자적으로 검증한다.
- 동시 확인, 부분 입금, 역분개, 금액 불일치 테스트를 추가한다.
- 성공 조건: 어느 권한 운영자가 확인해도 주문 전체에 한 번만 반영되고 감사 기록이 남는다.

## P1 — 중앙 집하·보관·단일 출고

### P1-1. 중앙 출고지와 상품별 이력

- `fulfillment_centers`, 매장별 준비 업무, 상품별 이동/확인 이력을 추가한다.
- 초기 기본 중앙 출고지 한 곳을 등록한다.
- 성공 조건: 상품 위치가 매장 준비, 이동 중, 중앙 입고, 중앙 보관으로 추적된다.

### P1-2. 매장 준비 및 중앙 입고 화면

- 매장에는 자기 상품 준비·인계 목록을 제공한다.
- 중앙 담당자에게 주문별 입고 체크리스트와 문제 등록을 제공한다.
- 성공 조건: 매장의 “보냄”과 중앙의 “받음”이 별도 행위자로 기록된다.

### P1-3. 통합 출고 상태와 단일 송장 제약

- 모든 유효 상품의 중앙 입고·최종 확인·합포장을 송장 생성 선행 조건으로 만든다.
- 주문/배송 요청당 활성 Shipment와 송장을 하나로 제한한다.
- 매장 직접 발송 및 부분 발송 경로를 기본 차단한다.
- 성공 조건: 누락 상품이 있으면 송장을 만들 수 없고 정상 주문은 배송 1건·송장 1개만 가진다.

### P1-4. 중앙 보관 모델 통합

- 기존 `storage_expires_at`을 중앙 위치·입고 기록과 연결한다.
- 여러 주문의 보관 상품을 한 배송 요청으로 묶는다.
- 배송비 결제 중복 방지와 보관 만료 처리를 명시한다.
- 성공 조건: 보관 상품의 물리 위치와 단일 합배송이 추적된다.

## P2 — 운영 편의와 미래 확장

### P2-1. 운영 대시보드

- 공용 입금, 매장 준비, 중앙 미입고, 출고 보류를 역할별 작업 큐로 제공한다.
- 장기 미처리와 금액/상품 불일치를 알린다.

### P2-2. 감사·보고

- 입금, 상품 상태, 인계, 중앙 입고, 송장, 취소·환불 변경 이력을 검색 가능하게 한다.
- 매장별 처리 실적과 전체 사업체 매출을 분리해 보여준다.

### P2-3. PortOne 재활성화 준비 검증

- 현재는 활성화하지 않는다.
- 미래 어댑터 계약, 데이터 매핑, 웹훅 멱등성 테스트와 재활성화 체크리스트만 유지한다.
- 재활성화는 별도 제품 결정과 운영 검증을 거친다.

## 권장 브랜치/작업 순서

1. 문서 기준 확정 및 미결 정책 결정
2. `feature/manual-transfer-only`
3. `feature/store-memberships-permissions`
4. `feature/direct-product-publishing`
5. `feature/shared-transfer-operations`
6. `feature/central-fulfillment-model`
7. `feature/central-fulfillment-console`
8. `feature/single-shipment-constraint`
9. `feature/central-storage-consolidation`

브랜치 이름은 예시다. 각 브랜치는 독립 검토 가능한 작은 커밋으로 나누고, 앞 단계의 데이터 계약이 확정되기 전에 뒤 단계 UI를 먼저 만들지 않는다.
