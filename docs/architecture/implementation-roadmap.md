# 구현 로드맵

## 작업 원칙

- 이 문서 작성까지는 상거래 코드를 변경하지 않는다.
- 실제 구현은 현재 문서 변경과 분리된 새 브랜치에서 시작한다.
- 한 작업은 가능한 한 하나의 데이터 계약 또는 사용자 흐름만 변경한다.
- DB 변경은 새 순방향 마이그레이션으로 수행하고 적용된 기존 마이그레이션을 수정하지 않는다.
- 결제·권한·주문·재고·배송 변경은 관련 테스트, 린트, 빌드, 마이그레이션 정합성을 검증한다.
- 운영 환경과 인증이 필요한 흐름은 정적 검사와 별도로 실제 검증 결과를 기록한다.

## P0 — 기준과 거래 안전성

### P0-1. 미결 정책 결정 기록

- [현행 감사](./current-state-audit.md)의 7개 미결 항목을 결정한다.
- 결정마다 데이터/API/UI 영향과 승인 날짜를 기록한다.
- 성공 조건: 이후 스키마가 추측 없이 작성 가능하다.

### P0-2. 수동 계좌이체 단일 활성 경로

상태: 운영 DB와 Production 반영 완료. 수동이체만 활성화하고 PortOne 공개 실행 경로는 비활성 어댑터로 보존한다.

- 고객 UI에서 PortOne 선택과 실행 진입점을 제거한다.
- 서버 체크아웃은 수동 계좌이체만 허용한다.
- PortOne 라우트와 어댑터는 명시적 비활성 상태로 격리·보존한다.
- 운영 중 기능 설정이 우연히 PortOne으로 바뀌지 않도록 서버 제약을 둔다.
- 성공 조건: 수동 주문·입금 원장은 정상이고 PortOne 공개 경로는 실행되지 않으며 관련 코드는 재활성화 가능한 형태로 남는다.

### P0-3. 매장 멤버십과 세부 권한

- `store_memberships` 또는 동등 모델을 추가한다.
- 기존 운영자·직원 데이터를 무손실로 이행한다.
- API와 RLS가 같은 매장 범위를 검사하도록 한다.
- 성공 조건: 운영자와 직원은 허용된 매장만 다루고 권한 있는 직원의 상품 공개가 가능하다.

### P0-4. 상품 작성·즉시 공개 계약

- 외부 승인 의미를 제거한다.
- `draft` 저장과 권한 보유자의 즉시 `active` 공개를 구분한다.
- 검수 메모는 상품 설명 데이터로 유지하고 승인 상태와 분리한다.
- 성공 조건: 승인 대기자 없이 자기 매장 상품을 공개하며 타 매장 수정은 차단된다.

### P0-5. 통합 입금의 공동 조회·멱등 확인

상태: `20260721134000`~`20260721143000`과 큐 snapshot `20260722010000` 운영 DB 적용, 코드 `625942c` Production 배포에 이어 역분개 결박 `20260722020000` 운영 DB 적용과 코드 `fe88ae9` Vercel Production 배포 `dpl_8CnZjNxyCLFcW6WxAHf67Cyvp99j`까지 완료했다. 공개 검증과 인증된 Owner/운영자 읽기 화면 검증은 완료했지만 운영 원장이 비어 실제 입금·역분개 mutation은 미검증이며, 경매·배송비 원장 이력/정정 UI는 후속 P1이다.

- 확정된 공개 범위로 공용 입금 큐를 제공한다.
- 원장 기록과 주문 결제 확정을 원자적으로 검증한다.
- 동시 확인, 부분 입금, 역분개, 금액 불일치 테스트를 추가한다.
- 배포는 먼저 외부 직접 확정과 구 원장 mutation 권한을 차단한 뒤, 관련 행 잠금을 `NOWAIT`로 확보하고 기존 원장 무결성 검사를 통과한 경우에만 새 RPC 권한을 연다.
- 0단계 전에 운영자·Owner 입금 mutation 트래픽을 중지하고 실행 중 요청을 비운다. 분 단위 실행 경계를 피하고 `process-auction-purchase-offers`의 run/activity가 없을 때 `20260721134000`을 실행한다. 이 단계는 원장-상태 가드 트리거를 먼저 설치하고 기존 pg_cron의 job ID·이름·schedule·command·database·username·active 원상태를 비공개 snapshot으로 저장한 뒤 같은 job을 비활성화한다. 함수 `EXECUTE` 회수만으로 이 작업은 멈추지 않는다.
- `cron.log_run = on`을 먼저 확인한다. 0단계 커밋 뒤 최소 한 scheduler cycle을 두고 `cron.job.active = false`, `cron.job_run_details`에 `end_time is null`이면서 `starting`·`connecting`·`sending`·`running`인 해당 job 없음, `pg_stat_activity`의 실행 호출 없음과 조회 권한을 운영자가 확인한다. 서버 재시작 뒤 종료 시각 없이 남은 실패 이력은 실제 backend 부재를 확인한 후에만 별도로 정리하며, 자동 연속 적용만으로 드레인을 증명하지 않는다.
- `20260721135000`과 `20260721140000`은 cron이 비활성·드레인 상태가 아니거나 snapshot의 ID·metadata와 달라졌으면 실패한다. `20260721140000`은 모든 새 계약이 한 트랜잭션에서 성공한 마지막에만 같은 job의 원래 `active` 값을 복원하고 전체 metadata를 다시 대조한다. 따라서 원래 꺼져 있던 job을 임의로 켜지 않는다. 1단계 이후 2단계가 잠금 또는 무결성 검사에서 실패하면 구 mutation 권한과 자동 경매 정산은 닫힌 채로 유지되므로, 데이터를 검토한 뒤 모든 writer를 계속 중지한 상태에서 2단계를 재실행한다.
- 구 역분개 RPC는 확정된 구매 제안 연결 주문을 0원 입금 대기로 되돌리면서 제안의 `settled` 상태는 되돌리지 못했다. `20260721140000`은 이런 stranded 상태를 포함해 입금 대기 주문과 연결 제안 상태가 맞지 않으면 실패-폐쇄한다. 적용 전에 해당 행을 감사하고 Owner가 승인한 별도 reconciliation 절차로 정리하며 마이그레이션이 임의로 상태를 추정해 고치지 않는다.
- `20260721141000`은 고객 주문과 공용 계좌 입금요청을 한 RPC 트랜잭션으로 묶고, 인증 사용자의 구 주문 단독 생성 RPC 실행 권한을 회수한다. 앱은 이 마이그레이션까지 적용된 뒤 배포한다.
- `20260721142000`은 21~22시 KST 경매 mutation blackout을 유지하면서 체크아웃이 선택한 `active` 고정가 상품만 `closed`로 바꾸는 정확한 예외를 허용한다. 경매 상품이나 다른 필드 변경에는 예외가 없다.
- `20260721143000`은 새 Supabase 프로젝트의 Data API 기본값과 기존 프로젝트의 잔존 ACL 차이를 없애기 위해 service-key 서버가 직접 사용하는 테이블 권한을 명시적으로 재설정한다. 적용 직전 저장소 밖 배치·운영 도구에 숨은 service-key 직접 writer가 없는지 인벤토리하고, 발견하면 같은 최소 권한 표에 먼저 반영한다. 내부 확정/원장 RPC의 `service_role EXECUTE`는 계속 금지한다.
- 구매 제안 연결 경매와 배송 요청 연결 배송비의 역분개는 후속 재정산 상태 모델이 생기기 전까지 실패-폐쇄한다. 이미 확정 알림이 발송된 고정가 주문의 입금을 정정하면 반대 원장과 주문 재개뿐 아니라 고객 정정 알림도 같은 트랜잭션에서 남긴다.
- 서로 다른 운영자가 같은 시점의 입금 화면에서 기록하면, 서버는 부모 행 잠금 뒤 화면이 관찰한 전체 원장 누적액과 단조 증가 원장 행 수를 함께 비교한다. 둘 중 하나라도 달라졌으면 기록하지 않고 새로고침을 요구한다. 결과를 확인하지 못한 브라우저 재시도 키는 인증된 처리자별로 격리해 공유 단말의 계정 전환이 다른 처리자의 키를 소비하지 못하게 하고, 성공 응답은 신규 기록과 기존 멱등 영수증 재확인을 구분한다. 원장 합계·행 수와 주문별 상품 요약은 Data API 행 제한과 무관한 집계 RPC를 기준으로 하며, 운영자 입금 큐는 매장 소속과 무관하게 공용 통합 주문을 조회한다. 활성 큐는 안전 상한 400건을 넘으면 일부를 숨기지 않고 계속 실패-폐쇄한다. 운영 반영 P1은 완료 이력에 keyset/load-more를 추가하지만 활성 큐 자체의 상한 제거는 운영 용량·성능 계획 뒤 별도 작업이다.
- 경매의 첫 부분입금은 입금 기한 안에서만 받을 수 있다. 부분입금이 생긴 주문과 구매 제안은 같은 트랜잭션에서 자동 만료 기한을 보류하며, 현재 시각이나 변경 가능한 설정으로 재계산하지 않고 주문/제안의 정확한 원래 기한과 hold 시각을 보존한다. 기존 부분입금도 cron 재개 전에 같은 형식으로 backfill한다. 순원장이 0원으로 돌아온 구매 제안 미연결 주문은 저장 원값을 복원한다. 잘못 기록한 구매 제안 연결 입금의 역분개·보류 해제는 별도 Owner 재정산 상태 모델이 생기기 전까지 자동 처리하지 않는다.
- 숨은 Owner 테스트 회원은 Owner 전용 proxy로만 조작한다. 운영자 공용 경매 목록, 잔액 RPC, 입금 기록, 역분개는 같은 제외 조건을 강제한다.
- 격리 검증은 기존 64개 마이그레이션 재생 후 새 6개를 PostgreSQL 17.10에 적용해 수행했다. 별도 fixture에서 구 0원 역분개 stranded 상태의 실패-폐쇄·DDL rollback·cron 비활성 유지, 멱등 replay의 원장 버전 응답과 현재 매장 권한 재검사도 확인했다. 실제 Supabase/PostgREST, 운영 PostgreSQL patch 버전, pg_cron C 확장 worker, 인증 브라우저 세션을 사용하지 않았으므로 운영 검증과 구분한다.
- 운영 반영 P1: 완료된 고정가 주문은 요청 시각 100건 절단 대신 요청·확정·최신 원장 시각의 최댓값과 UUID를 묶은 cursor로 조회한다. 활성/완료 소속, CAS 합계·버전, 최근 원장은 한 SQL 문장 snapshot이지만 상품 요약 보강과 다음 cursor 호출은 별도 snapshot이다. live pagination 중 첫 cursor 위에 새로 생긴 완료 건은 새로고침으로 회수한다. 현재 `cancelled_at`이 없어 원장 없는 취소 건의 정확한 전이 시각은 후속 상태 모델에서 보강해야 한다. Owner의 경매·배송비 화면도 원장 ID·행위자·시각·사유 이력과 허용된 정정 동선을 제공해야 한다.
- 후속 P1의 역분개 안전화는 `20260722020000_harden_manual_transfer_reversal.sql`로 완료했다. 두 RPC는 URL transfer 종류·ID와 원본 ledger 관계를 같은 mutation 안에서 결박하고 canonical 부모 행 잠금 뒤 signed 원장 합계·행 수 CAS를 검사한다. 새 역분개는 처리자별 UUIDv4 멱등 키를 사용하며 같은 대상·원장·사유 replay만 같은 역분개 행으로 재확인하고, 앱은 `reversal_of`를 포함한 엄격한 결과 계약을 확인한 뒤에만 재시도 키를 제거한다.
- P1 배포 기록: 운영 큐·최근 ledger writer·충돌 잠금이 모두 0임을 사전 확인하고 DB 마이그레이션을 먼저 적용했다. 400/401 경계는 격리 PostgreSQL fixture에서 검증했으며, 운영 DB의 함수 권한·유효 인덱스 2개·`active_count=0`·`active_overflow=false`·`integrity_error=false`를 읽기 전용으로 확인한 뒤 앱을 배포했다. Vercel source dry-run에서 발견한 로컬 `.env`·`.dev.vars`·사용자 설정·`work/` 산출물은 `.vercelignore`로 차단하고 재검증했다. 앱 smoke에서 503 또는 지연이 발생하면 앱만 `06519e5`로 되돌리고 additive 함수·인덱스는 유지한 채 원인을 조사한다. live 운영 중 하향 마이그레이션이나 함수/인덱스 삭제로 롤백하지 않는다.
- 역분개 결박 P1 배포 기록: 운영 원장이 비어 있고 관련 writer·잠금이 없음을 확인한 뒤 `20260722020000`을 DB 먼저 적용하고 코드 `fe88ae9`를 Vercel Production 배포 `dpl_8CnZjNxyCLFcW6WxAHf67Cyvp99j`로 공개했다. PostgreSQL 18.4 격리 환경에서 계약과 실제 다중 세션 경쟁을 검증했다. 공개 통합 검사, `site/status` DB 연결, 비인증 원장 mutation 차단과 인증된 운영자 주문·대시보드 읽기도 재검증했다. Docker Desktop 설치와 Compose runner 작성은 완료했으나 Windows 재부팅 전에는 엔진을 시작할 수 없어 Docker smoke는 대기 중이다. 운영 데이터가 없어 실제 입금·역분개 mutation은 실행하지 않았다.
- 성공 조건: 어느 권한 운영자가 확인해도 주문 전체에 한 번만 반영되고 감사 기록이 남는다.

## P1 — 중앙 집하·보관·단일 출고

### P1-1. 중앙 출고지와 상품별 이력

- `fulfillment_centers`, 매장별 준비 업무, 상품별 이동/확인 이력을 추가한다.
- 초기 기본 중앙 출고지 한 곳을 등록한다.
- 성공 조건: 상품 위치가 매장 준비, 이동 중, 중앙 입고, 중앙 보관으로 추적된다.

### P1-2. 매장 준비 및 중앙 입고 화면

- 매장에는 자기 상품 준비·인계 목록을 제공한다.
- 중앙 담당자에게 주문별 입고 체크리스트와 문제 등록을 제공한다.
- 성공 조건: 매장의 “보냄”과 중앙의 “받음”이 별도 행위자로 기록된다.

### P1-3. 통합 출고 상태와 단일 송장 제약

- 모든 유효 상품의 중앙 입고·최종 확인·합포장을 송장 생성 선행 조건으로 만든다.
- 주문/배송 요청당 활성 Shipment와 송장을 하나로 제한한다.
- 매장 직접 발송 및 부분 발송 경로를 기본 차단한다.
- 성공 조건: 누락 상품이 있으면 송장을 만들 수 없고 정상 주문은 배송 1건·송장 1개만 가진다.

### P1-4. 중앙 보관 모델 통합

- 기존 `storage_expires_at`을 중앙 위치·입고 기록과 연결한다.
- 여러 주문의 보관 상품을 한 배송 요청으로 묶는다.
- 배송비 결제 중복 방지와 보관 만료 처리를 명시한다.
- 성공 조건: 보관 상품의 물리 위치와 단일 합배송이 추적된다.

## P2 — 운영 편의와 미래 확장

### P2-1. 운영 대시보드

- 공용 입금, 매장 준비, 중앙 미입고, 출고 보류를 역할별 작업 큐로 제공한다.
- 장기 미처리와 금액/상품 불일치를 알린다.

### P2-2. 감사·보고

- 입금, 상품 상태, 인계, 중앙 입고, 송장, 취소·환불 변경 이력을 검색 가능하게 한다.
- 매장별 처리 실적과 전체 사업체 매출을 분리해 보여준다.

### P2-3. PortOne 재활성화 준비 검증

- 현재는 활성화하지 않는다.
- 미래 어댑터 계약, 데이터 매핑, 웹훅 멱등성 테스트와 재활성화 체크리스트만 유지한다.
- 재활성화는 별도 제품 결정과 운영 검증을 거친다.
