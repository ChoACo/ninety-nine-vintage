# 주문·결제·중앙출고 흐름

## 고객 기준 정상 흐름

```text
여러 매장 상품을 통합 장바구니에 담음
→ 통합 주문 1건 생성
→ 공용 계좌 안내 및 1회 입금
→ 권한 있는 운영자 1명이 통합 입금 확인
→ 각 매장이 자기 상품 확인·준비
→ 각 매장이 중앙 출고지로 인계
→ B 중앙 출고지가 상품별 입고·보관 확인
→ 고객이 결제 완료 보관 상품을 선택해 배송 요청 1건 생성
→ 배송비 입금·확정
→ 요청 상품의 매장별 업무 완료와 B 보관 상태 확인
→ B에서 해당 배송 요청을 합포장
→ 배송 요청 1건·송장 1개 생성
→ 고객에게 발송
```

## 1. 장바구니와 주문 생성

- 장바구니는 고객 단위이며 매장별로 분리하지 않는다.
- 체크아웃 요청은 상품 ID 집합, 멱등 키, 서버가 확인할 결제 모드를 전달한다. 배송지와 배송비는 이 단계에서 받지 않는다.
- 서버는 상품 공개 상태, 재고/예약, 실제 가격, 결제 모드를 다시 확인한다.
- 하나의 트랜잭션에서 통합 주문과 주문 상품을 만든다.
- 주문 상품은 주문 시점의 `store_id`와 가격을 보존한다.

## 2. 수동 계좌이체

1. 통합 주문에 입금 안내 1건을 만든다.
2. 공용 은행명·계좌번호와 최종 금액을 스냅샷으로 저장한다.
3. 고객은 매장 수와 무관하게 총액을 한 번 입금한다.
4. 권한 있는 운영자는 공용 입금 목록을 조회한다.
5. 한 운영자가 실제 입금액과 입금자명을 원장에 기록한다.
6. 누적 유효 입금액이 예정액과 일치할 때만 주문 전체 결제를 확정한다.
7. 동시 확인과 재시도는 데이터베이스 잠금 및 멱등 규칙으로 중복 확정을 막는다.

입금 정정은 기존 기록을 덮어쓰지 않고 반대 원장 항목과 사유를 추가한다. 확인자, 정정자, 시각을 보존한다. 이미 결제 완료로 안내된 통합 주문을 다시 입금 대기로 여는 경우에는 같은 트랜잭션에서 고객 정정 알림도 추가한다.

경매 부분입금은 자동 승계가 돈을 받은 건을 만료시키지 않도록 현재 기한을 보류한다. 이때 주문 기한과 구매 제안 기한의 정확한 원값 및 hold 시각을 보존하며, 설정값과 현재 시각으로 원기한을 재계산하지 않는다. 구매 제안 미연결 건은 순원장이 0원이 되면 저장 원값을 복원하고, 구매 제안 연결 건은 Owner 전용 재정산 계약 전까지 역분개를 실패-폐쇄한다.

## 3. 매장별 준비

입금 확정 후 매장별 업무가 생성되거나 활성화된다.

```text
waiting_payment
→ preparing
→ ready_for_transfer
→ in_transit_to_center
→ center_received
```

- 운영자와 권한 있는 직원은 자기 매장 상품만 처리한다.
- `ready_for_transfer`는 상품 확인 완료이지 중앙 입고 완료가 아니다.
- 중앙 담당자가 실제 수량과 상품을 확인한 뒤에만 `center_received`가 된다.
- 상품별 처리자와 시각을 남긴다.

## 4. 중앙 집하와 보관

통합 주문의 중앙 상태는 다음처럼 진행한다.

```text
collecting
→ partially_received
→ all_received
→ checking
→ packing
→ ready_to_ship
→ shipped
→ delivered
→ completed
```

모든 유효 주문 상품이 중앙에 도착하지 않으면 `all_received` 이후 단계로 진행할 수 없다. 문제가 발견되면 상품을 `issue`로 표시하고 송장 생성을 막는다.

입금 확정은 상품별 `CustomerInventoryItem`을 정확히 한 번 생성한다. 이 레코드는 구매자의 보관 권리이며 실제 B 입고 사실과 구분한다. 중앙 입고와 `center_stored`가 기록된 뒤 상품은 발송 가능한 물리 상태가 된다. 고객은 여러 주문·경매·차순위 구매에서 생긴 보관 상품을 하나의 배송 요청으로 묶을 수 있다.

## 5. 단일 배송 제약

송장 생성 조건:

- 통합 결제 확인 완료
- 해당 배송 요청에서 제외되지 않은 모든 상품의 B 보관 완료
- 원등록 매장별 배송 업무 완료
- 배송지 확정
- 배송비 결제 확정 또는 명시적 면제
- 미해결 상품 예외 없음
- 합포장 완료
- 활성 송장이 아직 없음

기본 제품 정책은 매장 직접 발송과 한 배송 요청 내부의 부분 발송을 금지한다. 한 주문의 상품을 여러 배송 요청으로 나누는 것은 허용하지만, 한 상품은 진행 중인 요청 하나에만 포함되고 발송 완료 후 다시 신청할 수 없다. 예외가 필요해지면 기존 규칙을 우회하지 말고 제품 원칙 변경으로 먼저 승인한다.

## 6. 취소와 문제 처리

- 입금 전: 예약 해제와 주문 취소를 원자적으로 처리한다.
- 입금 후 상품 문제: 해당 상품을 보류하고 부분 취소 금액과 배송비 영향을 계산한다.
- 수동 환불: 환불 예정액, 계좌, 처리자, 완료 시각을 별도 원장에 남긴다.
- 일부 상품 취소 후에도 현재 배송 요청의 남은 상품은 B 보관·매장 업무·배송비·예외 조건을 모두 충족하기 전까지 발송하지 않는다.
- 전체 주문 상태는 상품 취소와 매장/중앙 상태를 반영해 서버가 결정한다.

## 7. PortOne 경계

활성 체크아웃은 수동 계좌이체 어댑터만 호출한다. PortOne 준비·동기화·웹훅 코드는 독립 어댑터 경계 뒤에 보존하고 라우트 노출, UI 선택, 런타임 모드 전환을 기본 비활성화한다. 향후 활성화 시에도 통합 주문, 주문 상품, 매장 준비, 중앙 출고 모델은 재사용한다.
